================================================================================
                            PROJECT SUMMARY
================================================================================

WHAT IS GAMEHUB:
A multiplayer game room platform for real-time gaming. Players create or join
rooms, select games, and play together. Features three visual themes and
real-time communication via Socket.io.

TECH STACK:
- Backend:  Express.js, Socket.io, MongoDB + Mongoose, UUID
- Auth:     bcryptjs, jsonwebtoken, passport, passport-google-oauth20, passport-discord
- Frontend: React 19, Vite (build tool), Tailwind CSS (CDN), QRCode.js (CDN)
- Dev:      Nodemon, Concurrently, npm
- DB:       MongoDB on Railway (shuttle.proxy.rlwy.net:17041)

HOW TO RUN:
  npm run dev          - Starts BOTH servers concurrently:
                         Express on :3000 (API + Socket.io)
                         Vite on :5173 (frontend with HMR)
                         Open http://localhost:5173 in browser

  npm run build        - Builds frontend to client/dist/
  npm start            - Runs Express serving built frontend on :3000

================================================================================
                         ARCHITECTURE DECISIONS
================================================================================

APP.JSX IS THE ORCHESTRATOR:
- ALL state declarations live in App.jsx (~80+ useState calls)
- ALL useEffects live in App.jsx (socket handlers, rejoin, QR, etc.)
- ALL handler functions live in App.jsx
- Header renders in App.jsx (NOT inside page components)
- Modals render in App.jsx alongside their parent pages
- Pages are purely presentational - they receive props and render UI
- This avoids needing Context/Redux (would be over-engineering)

HEADER RENDERING:
- Header is rendered once in App.jsx for all pages EXCEPT 'rejoining'
- Pages do NOT import or render Header themselves

PAGE RENDERING FLOW IN APP.JSX:
  if (page === 'rejoining') -> RejoiningPage (no Header)
  if (page === 'landing')   -> Header + LandingPage
  if (page === 'countdown') -> Header + CountdownPage
  if (page === 'game')      -> Header + GamePage
  default (page === 'room') -> Header + RoomPage + Modals (GameSelector,
                               GameDescription, AvatarPicker, PlayerProfile,
                               CharacterInfo, MasterTips, QRModal)

SOCKET.IO APPROACH:
- Single socket.js module exports the socket instance
- Imported by App.jsx and PictionaryGame.jsx
- Vite dev server proxies /socket.io to Express on port 3000

TAILWIND & QRCODE:
- Tailwind CSS stays as CDN link in client/index.html
- QRCode.js stays as CDN script tag, accessed via window.QRCode

CHARACTER SVG EXPORTS:
- CharacterSVGs.jsx exports as BOTH named and default
- PictionaryGame imports: { CharacterSVG } (named)
- AvatarPicker/RoomPage import: CharacterSVG (default)

--- client/game/PictionaryGame.jsx ---
~1250 lines. Full Pictionary game with multi-round, multi-winner-per-turn support:
  - Drawing canvas with color picker and brush controls
  - Non-passive touch event listeners prevent page scroll while drawing
  - Chat/guessing area: fixed to bottom on mobile (max 30vh), normal on md+
  - 60s game timer, 10s rules countdown
  - Server-authoritative timer: timerEndTimeRef stores absolute endTime from server;
    useEffect ticks every 250ms computing Math.ceil((endTime - Date.now()) / 1000)
    so all clients converge on the same second regardless of network latency
  - Flat scoring: 300 pts for both drawer and guesser per correct guess
  - Multiple winners possible per turn (drawer earns 300 pts per pick too)
  - Uses shared MusicButton component, registers startMusicRef for Header control
  - Server-driven pause: pauseGame/resumeGame events sync all clients
  - chatFrozen state disables guesser input during pause
  - guessCooldown state: 3-second cooldown after sending guess (prevents spam)
  - handleInputFocus: scrolls to top on mobile when keyboard appears
  - Visibility change handler: requests gameSync when returning to tab/app
  - Requests gameSync on mount for mid-game rejoin after page refresh
  - gameSync event handler: syncs timer state, handles noActiveGame redirect
  - playerDisconnected event handler: updates drawing order with offline status
  - Drawer picking overlay: non-drawers see countdown + message during pause
  - winnerPendingRef: race condition guard between pick and timer expiry
  - roomIdRef: keeps currentRoom.id in a ref to avoid stale closures
  - timerEndTimeRef: absolute ms timestamp from server, cleared on pause/round/game end
  - pauseCountdown state + interval for non-drawer overlay countdown
  - Rules modal, PickWinner modal, TimerEnd modal, EndGame confirm (all inline)
  - Kick/Skip confirm modal (master only) for player management
  - confirmAction state: { type: 'kick'|'skip'|'skipSelf', playerName }
  - correctGuesses state: running list of winner picks (name, word, pts), persists across rounds
  - Round transitions, winner/no-winner announcements, drawing order UI with scores
  - Drawing order shows offline players with grayscale avatar + OFFLINE badge
  - Drawing order shows Kick/Skip buttons for master
  - advanceRound skips disconnected drawers automatically
  - Correct Guesses section below Drawing Order (no avatars, shows name + pts + word)
  - Prominent word display: gradient banner above header for drawer
  - End Game button (master only) with confirmation modal

================================================================================
                          GAME RULES -- PICTIONARY
================================================================================

MULTI-ROUND GAME FLOW:
  1. Master clicks "Start Game" -> server computes drawing order, emits gameStarting
  2. 3-2-1 countdown (round 1 only)
  3. Server calls startRound -> emits gameStarted + yourWord to drawer
  4. Rules Modal appears with 10s countdown (every round)
  5. At countdown = 0: modal closes, 60s game timer starts, drawing enabled
  6. Drawer draws, guessers type guesses
  7. Drawer clicks "Pick Winner" -> emits pauseGame -> ALL players' timers pause,
     chat freezes for guessers, non-drawers see "Drawer is picking a winner!" overlay
  8. Drawer selects winner -> escalating pts each -> 2s announcement -> canvas clears ->
     new word assigned -> timer resumes -> continue drawing
  9. Drawer can pick multiple winners per turn (new word each time)
  10. If 60s timer expires with guesses: Timer End Modal with 10s pick timer at 50 pts
  11. If 60s timer expires with no guesses: auto noWinner
  12. After turn ends: 3s result display ("Round Finished! Next drawer starting in 3s...")
  13. Server advances: more rounds -> nextRound; all done -> gameEnded -> room page

ROLES:
  Drawer: sees word, draws on canvas, picks winners, cannot guess
  Guesser: watches drawing, types guesses, cannot draw

POINTS SYSTEM (FLAT):
  300 pts per correct guess for both drawer and guesser.
  Multiple winners possible per turn (drawer earns 300 pts per pick too).
  0 pts if no winner selected.

SERVER-DRIVEN PAUSE:
  Drawer clicks "Pick Winner" -> emits pauseGame { roomId, pickDuration: 5 }
  Server: paused=true, captures timerRemainingMs, clears timerEndTime,
    broadcasts gamePaused { pickDuration }
  All clients: timerEndTimeRef=null, timerPaused=true, chatFrozen=true
  Non-drawer clients: show overlay with countdown from pickDuration
  Server blocks gameGuess while paused
  Cancel pick: emits resumeGame -> server computes new endTime from timerRemainingMs,
    broadcasts gameResumed { endTime } -> all clients restore timerEndTimeRef and resume
  Pick timer expiry: auto-closes modal, emits resumeGame
  continueDrawing: server computes new endTime from timerRemainingMs,
    emits continueDrawing { nextPickValue, endTime }

RACE CONDITION GUARD:
  winnerPendingRef set to true when selectWinner is called.
  Prevents timer-expiry logic from firing while server processes the pick.
  Cleared on continueDrawing, roundResult, winnerPicked, nextRound events.

COUNTDOWN TIMERS:
  Rules Countdown (10s): drawing disabled, game timer paused until 0
  Game Timer (60s): server-authoritative — server sends absolute endTime (Date.now()+60000),
    client computes display via Math.ceil((endTime - Date.now()) / 1000) every 250ms.
    All clients converge on the same second regardless of network latency.
    Server stores timerEndTime on room.game; captures timerRemainingMs on pause,
    recomputes new endTime on resume/continueDrawing. Pulses red at <=10s.
  Pick Timer: 5s mid-game (Pick Winner), 10s at timer end, auto-expiry

END GAME EARLY (Master only):
  Master clicks "End" button in game header -> confirmation modal.
  All points reset to 0. Game ends without recording history.
  Server emits gameEnded with cancelled: true.

GAME HISTORY:
  Server compiles at game end: { game, timestamp, finalScores, roundScores }
  Stored in room.gameHistory array (in-memory, per room).
  Sent to clients via gameEnded and rejoinSuccess events.
  PlayerProfile shows per-player history: total pts, placement, round breakdown.
  roundScores groups by round and sums (player can earn multiple times per round).

DRAWING ORDER:
  Sorted by score desc, random tiebreakers. Each player draws once.
  UI shows Done/Drawing Now/Up Next states + player scores.

WORD LIST (30 words in src/sockets.js):
  ELEPHANT, PIZZA, RAINBOW, ROCKET, CASTLE, GUITAR, ROBOT, TREASURE, UNICORN,
  DRAGON, MOUNTAIN, AIRPLANE, BIRTHDAY, TELESCOPE, WATERFALL, DINOSAUR,
  BUTTERFLY, SNOWMAN, PIRATE, VOLCANO, BANANA, LIGHTHOUSE, SKATEBOARD,
  MERMAID, TORNADO, SUNFLOWER, SPACESHIP, CAMPFIRE, PENGUIN, JELLYFISH

VALIDATION (server-side):
  - Only drawer can drawLine, clearCanvas, winnerPicked, noWinner, pauseGame, resumeGame
  - Only master can endGameEarly, kickPlayer, skipPlayerTurn
  - Master cannot kick themselves
  - skipPlayerTurn only works on the current drawer
  - Player names unique per room (case-insensitive, checked on joinRoom)
  - Drawer disconnect during game -> marks player.connected=false, skips their turn
  - Disconnected players shown with offline indicator in drawing order
  - Guesses blocked while game is paused
  - 3-second guess cooldown enforced client-side (prevents spam)
  - Avatar selection timeout: 30s to select non-meta avatar or get removed
  - kickPlayer/skipPlayerTurn emit error messages on validation failure (not silent)

VISIBILITY CHANGE HANDLING:

  - Client listens for 'visibilitychange' event

================================================================================
                          GAME RULES -- TRIVIA
================================================================================

SIMULTANEOUS-PLAY GAME FLOW:
  1. Master clicks "Start Game" with Trivia Master selected -> gameStarting with gameType='trivia'
  2. 3-2-1 countdown -> TriviaGame mounts
  3. Round 1 starts: Rules modal (10s) with full game explanation
  4. For each question in round:
     a. Server sends triviaQuestion with shuffled answers
     b. All players answer simultaneously (10s timer, 5s for speed round)
     c. triviaAnswerReceived updates UI showing who has answered
     d. Timer expires OR all answered -> triviaReveal (3s showing correct answer + points)
     e. Advance to next question
  5. After all questions: triviaRecap with animated question review + live leaderboard
  6. Master clicks "Next Round" -> startNextTriviaRound
  7. Rounds 2-3: Rules modal (3s) "Same rules! Get ready!"
  8. Round 4 (Speed): Rules modal (5s) big red pulsating "SPEED ROUND! EXTRA POINTS!"
  9. After Round 4: triviaFinalResults -> winner celebration -> return to room

GAME STRUCTURE:
  - 4 rounds total
  - Rounds 1-3: 5 questions each, 10 seconds per question, simultaneous play
  - Round 4 (Speed Round): Individual race format, each player answers independently
  - Answer choices: 6 per question (Easy through Genius), 4 for Super Easy/Very Easy, 2 for Yes/No
  - Kid-friendly pop culture questions across 6 categories

SCOREBOARD BEHAVIOR:
  - Question phase: FROZEN at previous round's final scores (uses previousRoundScores state)
  - Reveal phase: FROZEN (same as question phase, no updates or indicators)
  - Recap phase: STARTS with previousRoundScores, UPDATES as questions are revealed,
    ENDS with final scores that become next round's frozen scores

POINTS SYSTEM:
  BASE_POINTS = 100
  TIME_BONUS_MAX = 50       (bonus for fast answers)
  SPEED_MULTIPLIER = 2      (Round 4 only)

  Points = (BASE_POINTS + time_bonus) * multiplier
  Time bonus formula: TIME_BONUS_MAX * (1 - timeTaken / questionDuration)
  Example: Answer in 2s of 10s = 100 + 40 = 140 pts
  Example: Speed round fast answer = (100 + 50) * 2 = 300 pts

DYNAMIC RULES MODAL:
  - Round 1: 10 seconds, full rules explanation
    • 6 answer choices per question
    • 10 seconds to answer
    • Faster answers = more points
    • Round 4 is Speed Round with double points
  - Rounds 2-3: 3 seconds, simple "Same rules! Get ready!"
  - Round 4: 5 seconds, big red pulsating "SPEED ROUND! EXTRA POINTS! BE QUICK!!!"

SPEED ROUND (ROUND 4) - INDIVIDUAL RACE FORMAT:
  - Each player plays independently (not simultaneous)
  - 60 seconds total, answer as many questions as possible
  - Correct answer: 1s delay showing green confirmation, then next question
  - Wrong answer: immediate next question (no delay)
  - Race visualization: players move along track based on correct answers
  - Race winner = player with most correct answers (points as tiebreaker)

SPEED RECAP FLOW (after race ends):
  1. Race phase (3s): Shows race result overlay on track
     - "You won the race!" or "{name} wins the race!" + speed points earned
  2. Winner phase (4s): "Speed Round Results"
     - Shows race winner with crown, all players' speed points
  3. Leaderboard phase (2.5s): "Final Standings"
     - Shows scores BEFORE speed round was added
  4. Animating phase (2s): "Adding speed round points..."
     - Scores animate up as speed points are added
  5. Final phase (5s): "Game Winner!"
     - Shows overall game winner with celebration
     - All players show "+X from speed round" indicator
  6. Countdown (5s): "Returning to game room in X..."

ANIMATED RECAP (Master-Controlled, Simultaneous Difficulty Groups Display):
  - Master controls timing with two buttons:
    • "Reveal All Questions" - instantly shows all Q&A with player results
    • "Go to Next Round" - advances when master is ready (no auto-progression)
  - ALL difficulty groups' questions shown vertically at once (not sequentially)
  - Each group shows: [Difficulty] question text, correct answer, player results
  - Cascade player reveal animation (150ms stagger per player)
  - Coin pop animation (animate-coinPop) for correct answers with points
  - Cross animation (animate-wrongCross) for wrong answers
  - Player slide-in animation (animate-playerReveal) for cascade effect
  - Dynamic viewing time: 3s base + 1s per additional group (if not using Reveal All)
  - Leaderboard updates after player reveals complete (or instantly with Reveal All)
  - Non-masters see "Waiting for game master to continue..." when ready

ANSWER RANDOMIZATION:
  - Server shuffles answers using Fisher-Yates algorithm
  - shuffledCorrectIndex tracked for scoring validation
  - Prevents "correct answer is always A" pattern

================================================================================
                 LESSONS LEARNED -- FUTURE GAME DEVELOPMENT
================================================================================

This section captures critical patterns and gotchas discovered during Pictionary
and Trivia development. Follow these guidelines to avoid re-learning mistakes.

--------------------------------------------------------------------------------
SCOREBOARD & STANDINGS MANAGEMENT
--------------------------------------------------------------------------------

PROBLEM WE HIT:
  Trivia scoreboard was updating during question phase, revealing who got answers
  right before the reveal phase. Users complained multiple times about this.

SOLUTION PATTERN:
  1. Create a "frozen scores" state (e.g., previousRoundScores) that captures
     standings at specific moments (end of recap/reveal phases)
  2. Question/playing phases display FROZEN scores only
  3. Reveal/recap phases can show live updates with animations
  4. Save final scores to frozen state AFTER recap animation completes, not before

CODE PATTERN:
  const [previousRoundScores, setPreviousRoundScores] = useState({});

  // In recap completion handler (AFTER animation, not at start):
  const finalScores = {};
  standings.forEach(p => { finalScores[p.name] = p.score; });
  setPreviousRoundScores(finalScores);

  // In question/reveal phase scoreboard:
  .map(p => ({ ...p, displayScore: previousRoundScores[p.name] || 0 }))

RULE: Decide upfront for each game phase whether scoreboard is:
  - FROZEN (show previousRoundScores, no indicators)
  - UPDATING (show live standings with +points animations)
  - HIDDEN (don't show at all)

--------------------------------------------------------------------------------
PHASE TRANSITIONS & USER COMPREHENSION TIME
--------------------------------------------------------------------------------

PROBLEM WE HIT:
  Phases transitioning too fast. Users couldn't see who won, what happened,
  or process results before the next thing appeared.

SOLUTION PATTERN:
  1. Add deliberate delays between phases (2-5 seconds typical)
  2. Show clear announcements/celebrations before moving on
  3. Use multi-phase recaps for complex results (race winner vs game winner)
  4. Let animations COMPLETE before advancing

TIMING CONSTANTS TO DEFINE UPFRONT:
  const PHASE_TIMINGS = {
    RULES_DISPLAY: 10000,        // How long to show rules
    QUESTION_DURATION: 10000,    // Time to answer
    REVEAL_DURATION: 3000,       // Time to show correct answer
    RECAP_QUESTION_DELAY: 2000,  // Delay between recap questions
    WINNER_CELEBRATION: 4000,    // Time to celebrate winner
    PHASE_TRANSITION: 1000,      // Buffer between phases
    PRE_ANIMATION_PAUSE: 500,    // Pause before animation starts
    POST_ANIMATION_PAUSE: 1000,  // Pause after animation ends
  };

RULE: When in doubt, make transitions SLOWER. Fast feels rushed; slow feels
polished. Users need time to read, react, and celebrate.

--------------------------------------------------------------------------------
MULTI-PHASE RECAP FLOWS
--------------------------------------------------------------------------------

PROBLEM WE HIT:
  Speed round had race winner AND game winner. Showing both at once was confusing.
  Users didn't understand which winner was which.

SOLUTION PATTERN:
  Create distinct phases with clear headers for each piece of information:

  TRIVIA SPEED RECAP EXAMPLE:
    Phase 1: 'race'       - Show race finishing, overlay "X wins the race!"
    Phase 2: 'winner'     - "Speed Round Results" - show race winner, all points
    Phase 3: 'leaderboard'- "Final Standings" - show pre-speed scores
    Phase 4: 'animating'  - "Adding speed round points..." - animate score changes
    Phase 5: 'final'      - "Game Winner!" - crown the overall winner

  Each phase has:
    - Clear header text explaining what's being shown
    - Sufficient display time (2-5 seconds)
    - Visual distinction (colors, icons, animations)

RULE: If showing multiple types of "winners" or "results", separate them into
distinct phases with different headers. Never assume users will figure it out.

--------------------------------------------------------------------------------
ANIMATION PATTERNS
--------------------------------------------------------------------------------

PHASED ANIMATIONS (from Trivia/QuickMath recap):
  Phase 0: Show ALL difficulty groups' questions simultaneously (1s)
  Phase 1: Cascade reveal players with 150ms stagger (duration varies)
    - Correct: green + coin pop animation (animate-coinPop)
    - Wrong: red + cross animation (animate-wrongCross)
    - All: slide-in animation (animate-playerReveal)
  Phase 2: Animate points from 0 to earned value (0.8s)
  Phase 3: Update standings, dynamic viewing time (3s base + 1s per group)

  This creates anticipation and lets users follow what's happening.

CSS ANIMATION CLASSES (in client/index.html):
  - animate-wrongCross: scale 0 → 1.3 → 1.0 with rotation, 0.5s
  - animate-coinPop: scale 0 → 1.2 → 1.0, 0.4s
  - animate-playerReveal: slide-in from left (-20px → 0), 0.3s

FLYING POINTS PATTERN:
  const [flyingPoints, setFlyingPoints] = useState([]);
  // Show "+100" floating above player briefly
  setFlyingPoints([{ playerName: 'Alice', points: 100 }]);
  setTimeout(() => setFlyingPoints([]), 500);

SCORE COUNTER ANIMATION:
  // Animate score from old value to new value over duration
  const animDuration = 800;
  const steps = 20;
  let step = 0;
  const interval = setInterval(() => {
    step++;
    const progress = step / steps;
    const easedProgress = 1 - Math.pow(1 - progress, 3); // ease-out
    setAnimatedScore(Math.round(oldScore + (pointsEarned * easedProgress)));
    if (step >= steps) clearInterval(interval);
  }, animDuration / steps);

RULE: Animate score changes, don't just update them. Users notice and appreciate
the visual feedback.

--------------------------------------------------------------------------------
CORRECT ANSWER FEEDBACK
--------------------------------------------------------------------------------

PROBLEM WE HIT:
  Speed round correct answers felt instant and unconfirming. Users weren't sure
  if they got it right before the next question appeared.

SOLUTION PATTERN:
  1. Show visual confirmation (green highlight, checkmark, "Correct!")
  2. Add brief delay (1 second) before next action
  3. Server tracks "isWaiting" state during delay
  4. Client shows confirmation UI during wait

SERVER PATTERN:
  if (isCorrect) {
    progress.isWaiting = true;
    io.to(player.socketId).emit('correctAnswer', { points });
    setTimeout(() => {
      progress.isWaiting = false;
      sendNextQuestion(player);
    }, CORRECT_DELAY_MS);
  }

CLIENT PATTERN:
  {isWaiting && (
    <div className="bg-green-500/30 border-2 border-green-500 p-4 rounded-xl">
      <div className="text-green-400 text-xl font-bold">✓ Correct!</div>
    </div>
  )}

RULE: Every correct answer needs visual confirmation + brief pause.

--------------------------------------------------------------------------------
MASTER CONTROLS DURING GAMES
--------------------------------------------------------------------------------

PROBLEM WE HIT:
  Kick/End Game buttons only appeared in certain phases, making them inaccessible
  when needed (e.g., during question phase when a player is AFK).

SOLUTION PATTERN:
  1. Create reusable GameMasterControls component
  2. Include in ALL game phases where master might need control
  3. Use consistent positioning (usually in sidebar/scoreboard area)

GAMEMASTERCONTROLS COMPONENT:
  - Kick Player button (opens player selection modal)
  - End Game button (opens confirmation modal)
  - Receives: theme, currentTheme, currentRoom, isMaster, playerName, availableCharacters
  - Returns null if !isMaster

RULE: Master controls should be accessible during ANY game phase, not just
specific moments. Place them in a consistent location across all phases.

--------------------------------------------------------------------------------
RECONNECTION & GAME SYNC
--------------------------------------------------------------------------------

PROBLEM WE HIT:
  Players refreshing mid-game or switching tabs would see stale/frozen UI.

SOLUTION PATTERN:
  1. On mount: emit requestGameSync { roomId }
  2. On visibility change (returning to tab): emit requestGameSync
  3. Server responds with FULL game state for current phase
  4. Client has handler to restore ALL relevant state

GAME SYNC RESPONSE MUST INCLUDE:
  - Current phase
  - Timer end time (absolute timestamp)
  - Current round/question data
  - Player's answer status (hasAnswered, selectedAnswer)
  - Standings/scores
  - Any phase-specific state (recap index, speed round progress, etc.)

RULE: Design the gameSync event EARLY. List all state that a reconnecting
player needs to fully restore their view.

--------------------------------------------------------------------------------
USEEFFECT DEPENDENCY MANAGEMENT
--------------------------------------------------------------------------------

PROBLEM WE HIT:
  Timer cleanup running prematurely when state changed, causing race conditions.
  Recap animation resetting mid-animation.

SOLUTION PATTERN:
  1. Separate useEffects by concern (timer tick vs phase transition vs animation)
  2. Use early returns for guard conditions
  3. Be explicit about which state triggers the effect

BAD PATTERN:
  useEffect(() => {
    // Timer + animation + phase transition all in one
    // Dependencies: [phase, timer, raceComplete, animationStep, ...]
  }, [phase, timer, raceComplete, animationStep]);

GOOD PATTERN:
  // Timer tick effect
  useEffect(() => {
    if (phase !== 'question') return;
    const interval = setInterval(...);
    return () => clearInterval(interval);
  }, [phase]);

  // Race completion effect (separate)
  useEffect(() => {
    if (phase !== 'speedRecap' || !raceComplete) return;
    const timer = setTimeout(() => setNextPhase('winner'), 2000);
    return () => clearTimeout(timer);
  }, [phase, raceComplete]);

RULE: One useEffect per concern. If you have more than 3-4 dependencies,
you're probably mixing concerns.

--------------------------------------------------------------------------------
"ANSWERED" INDICATORS & PLAYER STATUS
--------------------------------------------------------------------------------

PROBLEM WE HIT:
  "Answered" section disappeared between question and reveal phases, breaking
  continuity of what players had done.

SOLUTION PATTERN:
  1. Track answeredPlayers state that persists across related phases
  2. Show consistently in both question AND reveal phases
  3. Clear only when moving to genuinely new state (next question, new round)

PLAYER STATUS INDICATORS TO CONSIDER:
  - Waiting to answer (default)
  - Has answered (checkmark, no reveal of correctness)
  - Answered correctly (green, shown only in reveal/recap)
  - Answered incorrectly (red, shown only in reveal/recap)
  - Timed out / didn't answer (gray)
  - Disconnected (offline badge)

RULE: Decide upfront which indicators show in which phases. Create a matrix:

  | Indicator        | Question | Reveal | Recap |
  |------------------|----------|--------|-------|
  | Has answered     | ✓        | ✓      | -     |
  | Correct          | -        | ✓      | ✓     |
  | Wrong            | -        | ✓      | ✓     |
  | Points earned    | -        | -      | ✓     |

--------------------------------------------------------------------------------

PLAN THIS UPFRONT for each new game:

MODAL PATTERN -- ALWAYS FOLLOW:
  1. Modal component lives in client/modals/<ModalName>.jsx
  2. Modal state (object|null or boolean) lives in App.jsx
  3. Modal renders in App.jsx alongside its parent page, NOT inside the page
  4. Page components only receive the state SETTER to trigger the modal
  5. Modal imports its own dependencies (icons, data, etc.)
  EXCEPTION: PictionaryGame.jsx has inline modals (Rules, PickWinner, TimerEnd,
  EndGameConfirm, DrawerPickingOverlay) because they depend heavily on game state.
  EXCEPTION: LandingPage.jsx has inline character detail modal (showcaseChar state)
  because it's a simple browse-only popup with no external dependencies.

PAGE SIZE AWARENESS:
  - PictionaryGame.jsx (~1000 lines) and RoomPage.jsx (~820 lines) are largest
  - If a new feature needs >30 lines of JSX, extract to a modal or component

PLAYER IDENTIFICATION:
  - Server: socket.id | Client: player.name (unique per room)
  - Removing name uniqueness would require switching to socketId everywhere

PROP PASSING PATTERN:
  - Pages receive ALL data via props from App.jsx
  - Pages get SETTER functions to trigger modals
  - Pages do NOT get modal STATE if they don't need to read it
  - isMaster threaded: App.jsx -> GamePage -> PictionaryGame/TriviaGame
  - gameType threaded: App.jsx -> GamePage (for routing)
  - gameHistory threaded: App.jsx -> PlayerProfile

THEME-SPECIFIC STYLING:
  - All new UI must support all 3 themes in the JSX
  - TRON ONLY rule: focus on tron for MVP, polish others later
  - Pattern: theme === 'tron' ? ... : theme === 'kids' ? ... : ...

SOCKET.OFF PATTERN -- CRITICAL:
  NEVER use socket.off('eventName') without a handler reference. This removes
  ALL listeners including those from other components (e.g. PictionaryGame).
  ALWAYS use named handlers: socket.off('event', handler).
  App.jsx socket useEffect uses dependency [] with functional state updates.
  PictionaryGame socket useEffect uses dependency [playerName].

DEV DATA PATTERN (PlayerProfile):
  Dummy data constants defined inside PlayerProfile.jsx (not a data file).
  Will be replaced with real server data later -- do NOT architect this until asked.

GAME HISTORY DATA STRUCTURE:
  Server compiles at game end and stores in room.gameHistory:
  {
    game: "Drawing Game",
    timestamp: Date.now(),
    finalScores: [{ name, avatar, score }],  // sorted by score desc
    roundScores: { "PlayerA": [{ round: 1, points: 100 }, ...], ... }
  }
  A player can earn points multiple times in the same round (multiple picks).
  PlayerProfile groups by round number and sums when displaying.

MUSIC CONTROL PATTERN (Global Audio):
  - Global audio element lives in App.jsx, rendered for ALL pages (including landing)
  - musicStarted state + selectedTrack state + audioRef + isMuted live in App.jsx
  - Audio element uses muted={isMuted} attribute for muting
  - Header receives musicStarted, selectedTrack, onSelectTrack, onStartMusic, isMuted props
  - Header has separate mute button + music track dropdown (split controls)
  - Mute button: toggles isMuted, pulses when muted to indicate no sound
  - Music button: opens track selection dropdown with checkmark on current track
  - Auto-unmute: selecting a track calls onStartMusic() if not started, sets isMuted=false
  - Header music dropdown closes on click-outside (useEffect with ref)
  - playMusic() function in App.jsx calls play() only (no load() - browser auto-loads on src change)
  - onStartMusic callback calls playMusic() and sets musicStarted=true
  - useEffect in App.jsx listens for 'canplay' event when selectedTrack changes (prevents AbortError)
  - renderPageContent() helper function allows single return with global audio
  - Music persists across page navigation (audio element never unmounts)
  - No audio elements in RoomPage or PictionaryGame (removed)
  - No auto-play on room creation (browsers block autoplay without user interaction)

AFK STATUS TRACKING:
  - AFK useEffect in App.jsx tracks when player navigates away from room/game pages
  - Uses document.hidden (Visibility API) and page state to determine AFK
  - Emits setAfkStatus { roomId, isAfk } to server on visibility change or page change
  - CRITICAL: Captures roomId at effect start to avoid stale closure issues
  - Cleanup function emits AFK=true when navigating away (before currentRoom clears)
  - Server stores isAfk on player object, broadcasts playerAfkChanged to room
  - App.jsx listens for playerAfkChanged and updates currentRoom.players

CLOCK SYNC PATTERN (Timer Synchronization):
  - Client socket.js calculates clockOffset on connect/reconnect
  - Takes 5 samples of round-trip time, uses median offset for accuracy
  - Server responds to 'requestServerTime' with serverTime + clientSendTime echo
  - Offset = estimated server time - local time (accounts for network latency)
  - getServerTime() export: Date.now() + clockOffset (used by all game timers)
  - Games calculate remaining time: Math.ceil((endTime - getServerTime()) / 1000)
  - Eliminates cross-client timer drift caused by different device clocks

RECONNECTION SYSTEM (Multi-Layer):
  - Layer 1: 3-minute grace period (player stays in room with connected=false)
  - Layer 2: 10-minute expelled cache (player removed but can rejoin with full score + isMaster status)
  - Layer 3: localStorage persistence (survives browser close, provides rejoin UI)
  - Rejoin tokens: UUID per player, prevents impersonation on rejoin
  - Server expelledPlayers Map stores: playerData (including isMaster), roomId, timer, expelledAt
  - On rejoin: checks room first, then expelled cache, restores score and isMaster if found
  - Header rejoin button: appears when player has pendingSession and is not in room/game page
  - Clean serializable room data: roomCreated/roomJoined/rejoinSuccess emit clean copies to avoid circular refs

GAME-SPECIFIC RECONNECTION:
  - Pictionary: Drawer disconnect PAUSES game, stores remaining time, resumes on reconnect
  - Trivia/QuickMath: Disconnect during question phase checks if all remaining players answered
  - Trivia/QuickMath: Re-adds rejoining players to difficultyGroups (fixes "spectator" bug)
  - All games: Error handling (try-catch) around disconnect/reconnect code prevents server crashes
  - All games: Null checks on advanceToNextGroup/advanceRound functions prevent crashes
  - gameType set from room.game.gameType on rejoinSuccess (fixes wrong game loading after refresh)

MANUAL KICK POLICY (no auto-kick):
  - Disconnected players stay in room/game until master manually kicks them
  - Players marked player.connected=false, player.disconnectedAt=timestamp
  - Server emits 'playerDisconnected' to show AFK status in UI
  - Master can kick players using kick button (GameMasterControls or room lobby)
  - Kicked players stored in expelledPlayers cache (10-min rejoin window)
  - Room cleanup only happens when ALL players disconnected for 30+ minutes

AVATAR SELECTION TIMEOUT:
  - Server tracks avatarSelectionTimers Map (roomId:playerName -> timer)
  - 2-minute timeout starts when player joins with 'meta' avatar
  - Timer cancelled when player selects a non-meta avatar via changeAvatar
  - If timeout fires: player removed, 'removedForNoAvatar' sent to that player
  - If master times out: room is deleted since master is required

MASTER ROOM TRACKING:
  - Server tracks masterRooms Map (masterName -> roomId)
  - When master creates a new room: old room is closed with 'roomClosed' event
  - When master joins another room: old room is closed
  - Prevents orphaned rooms and ensures clean room lifecycle

SCORE ACCUMULATION (Room Session):
  - Player scores in room lobby accumulate across ALL games played in session
  - App.jsx onGameEnded: adds finalScores to existing p.score (not replaces)
  - Code: return updated ? { ...p, score: (p.score || 0) + updated.score } : p
  - Server endGameEarly: does NOT reset scores (preserves accumulated totals)
  - Cancelled games add 0 points (finalScores: [] sent to client)
  - Room scoreboard shows combined lifetime score for session

RECONNECTING PAGE TIMEOUT:
  - RejoiningPage.jsx has 5-second timeout via useEffect
  - If rejoin doesn't succeed within 5s, onTimeout callback fires
  - App.jsx clears currentRoom and navigates to landing page
  - Prevents players from being stuck on reconnecting screen forever

                    CREATING NEW GAMES -- ARCHITECTURE GUIDE
================================================================================

This section provides a comprehensive guide for creating new games in GameHub.
Follow these patterns to ensure consistency with existing architecture.

--------------------------------------------------------------------------------
GAME ARCHITECTURE OVERVIEW
--------------------------------------------------------------------------------

FILE STRUCTURE FOR A NEW GAME:
  client/game/
    |-- PictionaryGame.jsx      (existing - turn-based, ~1260 lines)
    |-- TriviaGame.jsx          (existing - simultaneous-play, ~2700 lines)
    |-- QuickMathGame.jsx       (existing - simultaneous-play with NumberPad, ~1850 lines)
    |-- NewGame.jsx             (your new game component)

  client/data/
    |-- games.js                (add game to MOCK_GAMES with gameType field)
    |-- triviaQuestions.js      (example of game-specific data file)

  src/sockets.js                (add game-specific socket handlers)

COMPONENT HIERARCHY:
  App.jsx
    └── GamePage.jsx (router - selects game component based on gameType)
        ├── PictionaryGame.jsx (turn-based drawing game)
        ├── TriviaGame.jsx (simultaneous-play quiz game)
        └── QuickMathGame.jsx (simultaneous-play math game with NumberPad)

KEY PRINCIPLE: GamePage.jsx routes based on gameType prop from App.jsx.
App.jsx captures gameType from gameStarting event and passes it down.
Games receive common props (theme, playerName, currentRoom, isMaster, etc.).

--------------------------------------------------------------------------------
GAME STATE MANAGEMENT (SERVER-SIDE)
--------------------------------------------------------------------------------

All game state lives on the server in room.game object. Pattern:

  room.game = {
    // Core state
    currentRound: 1,
    totalRounds: players.length,
    gameType: 'pictionary',           // or 'trivia', 'charades', etc.

    // Turn/player tracking
    drawingOrder: [...],              // array of { name, avatar, connected, done }
    currentDrawerIndex: 0,

    // Timer state (server-authoritative)
    timerEndTime: Date.now() + 60000, // absolute timestamp
    timerRemainingMs: null,           // captured when paused
    paused: false,

    // Game-specific state
    currentWord: 'ELEPHANT',          // pictionary-specific
    usedWords: ['ELEPHANT'],
    roundScores: {},                  // { playerName: [{ round, points }] }
    currentPickValue: 100,            // escalating scoring
  }

SERVER-AUTHORITATIVE PRINCIPLE:
  - Server owns ALL game state
  - Clients receive state via socket events and render it
  - Clients request state changes via emits, server validates and broadcasts
  - Never trust client-side computed state for scoring/rules

--------------------------------------------------------------------------------
SOCKET EVENT PATTERNS FOR GAMES
--------------------------------------------------------------------------------

STANDARD GAME EVENTS (use these patterns):

1. GAME START SEQUENCE:
   Client: startGame { roomId }
   Server:
     - Validates (master only, games selected, enough players)
     - Resets all player scores to 0 (all game types: Pictionary, Trivia, Quick Math)
     - Initializes room.game object
     - Emits gameStarting { roomId, drawingOrder, totalRounds, currentRound }
     - Client shows countdown page
     - After countdown: emits gameStarted { drawerName, currentRound, ... }
     - Emits yourWord { word } to drawer only (if applicable)

2. GAME TIMER PATTERN (server-authoritative):
   Server: gameTimerStart { endTime }    // absolute ms timestamp
   Client:
     - Stores endTime in ref (timerEndTimeRef.current = endTime)
     - useEffect ticks every 250ms: Math.ceil((endTime - Date.now()) / 1000)
     - All clients converge on same second regardless of latency

   PAUSE/RESUME:
   Client: pauseGame { roomId, pickDuration }
   Server:
     - Captures timerRemainingMs = timerEndTime - Date.now()
     - Clears timerEndTime
     - Emits gamePaused { pickDuration }

   Client: resumeGame { roomId }
   Server:
     - Computes new endTime = Date.now() + timerRemainingMs
     - Emits gameResumed { endTime }

3. PLAYER ACTION VALIDATION:
   Server validates ALL actions:
     - Is player allowed? (drawer vs guesser, master vs non-master)
     - Is game in valid state? (not paused, not ended)
     - Is action valid? (can't pick yourself as winner, etc.)

   On validation failure: socket.emit('error', { message: '...' })
   On success: broadcast to room via io.to(roomId).emit(...)

4. ROUND TRANSITION:
   Server: roundResult { winnerName, points, currentRound, totalRounds }
   Server: (after delay) nextRound { drawerName, currentRound, totalRounds }
   Client: updates UI, shows announcements, resets round state

5. GAME END:
   Server: gameEnded { finalScores, gameHistory, cancelled? }
   - finalScores: array sorted by score desc
   - gameHistory: full history object for PlayerProfile
   - cancelled: true if ended early (scores reset, game stays in queue)
   - Server removes completed game from room.selectedGames (unless cancelled)
   - This keeps server/client queue in sync for "Start Next Game"
   Client: navigates back to room page, shows final scores
   - Completed games shown greyed out in queue with "COMPLETED" label
   - Click completed game to view scoreboard with round breakdown
   - "Play Again" button re-adds game to queue

--------------------------------------------------------------------------------
CLIENT-SIDE GAME COMPONENT STRUCTURE
--------------------------------------------------------------------------------

RECOMMENDED STRUCTURE (see PictionaryGame.jsx):

```jsx
function NewGame({
  theme, currentTheme, playerName, selectedAvatar, availableCharacters,
  currentRoom, isMuted, isMaster, drawingOrder, currentRound, totalRounds
}) {
  // === STATE ===
  // Timer state
  const [gameTimer, setGameTimer] = useState(60);
  const timerEndTimeRef = useRef(null);

  // Game state
  const [isMyTurn, setIsMyTurn] = useState(false);
  const [currentPlayer, setCurrentPlayer] = useState('');

  // UI state
  const [showRulesModal, setShowRulesModal] = useState(true);
  const [chatMessages, setChatMessages] = useState([]);

  // Refs for closure safety
  const roomIdRef = useRef(currentRoom?.id);

  // === SOCKET EFFECT ===
  useEffect(() => {
    const handleGameStarted = (data) => { ... };
    const handleTimerStart = (data) => { ... };
    // ... more handlers

    socket.on('gameStarted', handleGameStarted);
    socket.on('gameTimerStart', handleTimerStart);

    // CRITICAL: Use named handlers for cleanup
    return () => {
      socket.off('gameStarted', handleGameStarted);
      socket.off('gameTimerStart', handleTimerStart);
    };
  }, [playerName]); // Minimal dependencies, use refs for room data

  // === TIMER EFFECT ===
  useEffect(() => {
    const interval = setInterval(() => {
      if (timerEndTimeRef.current) {
        const remaining = Math.ceil((timerEndTimeRef.current - Date.now()) / 1000);
        setGameTimer(Math.max(0, remaining));
        if (remaining <= 0) {
          // Timer expired logic
        }
      }
    }, 250);
    return () => clearInterval(interval);
  }, []);

  // === RENDER ===
  return (
    <div className="min-h-screen ...">
      {/* Game Header: round, score, timer */}
      {/* Game Content: varies by game type */}
      {/* Action Area: chat, buttons */}
      {/* Modals: rules, pick winner, etc. */}
    </div>
  );
}
```

KEY PATTERNS TO FOLLOW:

1. USE REFS FOR SOCKET CLOSURES:
   - roomIdRef.current = currentRoom?.id in effect, not currentRoom directly
   - Prevents stale closure issues when emitting from event handlers

2. NAMED EVENT HANDLERS:
   - Define handlers inside useEffect
   - ALWAYS socket.off('event', handler) with same reference
   - Never socket.off('event') without handler (removes all listeners)

3. TIMER DISPLAY:
   - Store absolute endTime from server in ref
   - Compute display value every 250ms
   - Never trust client-side countdown (clock drift, background tabs)

4. MODAL PATTERN:
   - Rules modal: shown at round start, countdown overlay, dismiss button
   - Action modal: pick winner, confirm actions, timer badge
   - All modals should handle game continuing in background

--------------------------------------------------------------------------------
SCORING SYSTEM PATTERNS
--------------------------------------------------------------------------------

FLAT SCORING (Pictionary pattern):
  - 300 pts per correct guess for both drawer and guesser
  - Multiple winners per turn: drawer earns 300 pts per pick
  - No winner: 0 pts

FIXED SCORING (simple pattern):
  - Correct answer: fixed points
  - Wrong answer: 0 or negative
  - Time bonus: extra points for speed

ROUND SCORES STRUCTURE:
  roundScores: {
    "PlayerA": [
      { round: 1, points: 100 },
      { round: 1, points: 200 },  // same player can earn multiple times
      { round: 2, points: 100 }
    ],
    "PlayerB": [...]
  }

FINAL SCORES:
  - Computed from player.score on each player object
  - Sorted desc for finalScores array in gameEnded

--------------------------------------------------------------------------------
TURN-BASED GAME PATTERNS
--------------------------------------------------------------------------------

DRAWING ORDER:
  - Computed at game start: computeDrawingOrder(players)
  - Sorted by score desc, random tiebreakers
  - Each player gets one turn (totalRounds = players.length)
  - drawingOrder[currentDrawerIndex] is current player

TURN STATE:
  - player.done: boolean, has this player completed their turn
  - room.game.currentDrawerIndex: which player in order
  - Server advances via advanceRound() function

SKIP/DISCONNECT HANDLING:
  - Disconnected players: marked connected=false, turn skipped
  - Master skip: emits skipPlayerTurn, server advances
  - advanceRound() checks player.connected, skips offline

--------------------------------------------------------------------------------
MULTI-ROOM HANDLING
--------------------------------------------------------------------------------

ROOM ISOLATION:
  - Each room has its own game state in rooms Map
  - Socket.io rooms: socket.join(roomId), io.to(roomId).emit(...)
  - Events scoped to room: always include roomId in payload
  - Never broadcast to all sockets, always to specific room

CONCURRENT GAMES:
  - Multiple rooms can play same or different games simultaneously
  - No shared state between rooms
  - Server handles each room independently

TRIVIA GAME IMPLEMENTATION (REFERENCE)
--------------------------------------------------------------------------------

The Trivia game demonstrates simultaneous-play architecture (vs turn-based Pictionary):

1. GAME TYPE ROUTING (games.js + GamePage.jsx):
   - games.js: Added gameType field ('trivia' or 'pictionary') to game entries
   - GamePage.jsx: Routes based on gameType prop from App.jsx
   - App.jsx: Captures gameType from gameStarting event, passes to GamePage

2. SERVER STATE STRUCTURE (sockets.js):
   room.game = {
     gameType: 'trivia',
     phase: 'rules' | 'question' | 'reveal' | 'recap' | 'finalRecap',
     currentQuestion: { ...question, shuffledAnswers, shuffledCorrectIndex },
     answers: { playerName: { answerIndex, timestamp } },
     questionHistory: [{ question, correctAnswer, playerResults }]
   }

3. ANSWER RANDOMIZATION:
   - shuffleArray() Fisher-Yates helper in sockets.js
   - advanceTriviaQuestion() shuffles before sending to clients
   - Stores shuffledCorrectIndex for validation in revealTriviaAnswer()

4. DYNAMIC RULES TIMING:
   - Three duration constants: ROUND1 (10s), NORMAL (3s), SPEED (5s)
   - startTriviaRound() selects duration based on currentRound
   - TriviaGame.jsx renderRulesPhase() shows different content per round type

5. ANIMATED RECAP (Simultaneous Difficulty Groups):
   - State: recapQuestionIndex, revealedPlayers, animatedStandings, previousRoundScores
   - ALL difficulty groups displayed vertically at once (not sequentially)
   - getActiveGroups() helper filters out empty difficulty groups
   - Cascade player reveal: 150ms stagger per player across all groups
   - CSS animations: animate-coinPop (correct), animate-wrongCross (wrong), animate-playerReveal
   - Dynamic viewing time: 3000ms base + 1000ms per additional group
   - Standings update after all player reveals complete
   - Auto-advances to next round after animation completes (no button needed)
   - previousRoundScores saved after recap completes for next round's frozen scoreboard

6. FROZEN SCOREBOARD (Question/Reveal phases):
   - previousRoundScores state holds scores from end of previous recap
   - Question phase: displays previousRoundScores (static, no updates)
   - Reveal phase: displays previousRoundScores (static, no indicators)
   - Only recap phase shows live updates to standings

7. SPEED ROUND RACE:
   - Individual play format (not simultaneous like rounds 1-3)
   - Server sends speedRoundQuestion events individually per player
   - Correct answer triggers 1s delay with green feedback before next question
   - Race visualization: shared track with player avatars moving based on correctCount
   - raceData contains: raceData (per-player stats), winner, maxCorrect

8. SPEED RECAP MULTI-PHASE FLOW:
   - speedRecapPhase state: 'race' → 'winner' → 'leaderboard' → 'animating' → 'final'
   - Race phase: overlay shows race winner on track (3s pause)
   - Winner phase: "Speed Round Results" with race winner celebration (4s)
   - Leaderboard phase: shows preSpeedStandings (scores before speed round) (2.5s)
   - Animating phase: animatedFinalScores interpolate from pre-speed to final (2s)
   - Final phase: "Game Winner!" with overall winner celebration (5s)
   - Clear visual distinction between race winner and game winner

9. KEY DIFFERENCES FROM PICTIONARY:
   - No drawer/guesser roles - all players equal (rounds 1-3)
   - No pause/resume - timer runs continuously
   - Phase-based flow instead of round-based
   - Auto-advance after recap animation (no master button for normal rounds)
   - Speed round uses individual race format with server-driven question delivery

MEMORY MASTER GAME ARCHITECTURE
================================================================================

Memory Master follows the simultaneous-play architecture (like Trivia/Quick Math) with 4 rounds:

FILES:
  - client/game/MemoryGame.jsx (~1700 lines) - Full Memory game component
  - client/data/memoryContent.js (~305 lines) - Items, configs, challenge types
  - src/sockets.js - Server-side game logic (memory* functions)

ROUND STRUCTURE (4 rounds, each with different challenge type):
  Round 1: Memory Match (find pairs)
  Round 2: Missing Item (identify removed item)
  Round 3: Spot the Difference (identify changed item)
  Round 4: Sequence Memory (speed round - recall sequence order)

CHALLENGE TYPES (CHALLENGE_TYPES constant):
  'match'      - Match pairs game (Round 1)
  'missing'    - Missing item identification (Round 2)
  'difference' - Spot the difference (Round 3)
  'sequence'   - Sequence recall - speed round (Round 4)

MATCH GAME (Round 1):
  - Grid sizes by difficulty (MATCH_GRID_CONFIG):
    super-easy: 2x2 (2 pairs)
    very-easy:  2x3 (3 pairs)
    easy:       3x2 (3 pairs)
    medium:     4x4 (8 pairs)
    hard:       4x4 (8 pairs)
    very-hard:  6x6 (18 pairs)
    genius:     6x6 (18 pairs)
  - 5 second memorization phase (cards shown face-up)
  - Cards flip face-down, 60 seconds to match all pairs
  - 50 points per matched pair
  - +50 bonus for perfect (no mistakes)
  - Client handles card flipping locally, submits results to server

DIFFICULTY GROUPS (same as Trivia/Quick Math):
  - Players grouped by difficulty setting
  - Each group sees content tailored to their difficulty
  - Grid sizes, display times, sequence lengths vary by difficulty

PHASE FLOW:
  rules -> display -> match/question -> reveal -> recap (repeats per round)
  Final: finalRecap with winner celebration

SOCKET EVENTS (Memory-specific):
  Client -> Server:
    memoryReady        { roomId }                    Player ready after rules
    memoryAnswer       { roomId, answer, ... }      Submit answer/match results
    requestMemorySync  { roomId }                   Request game state sync

  Server -> Client:
    memoryRulesStart   { round, challengeType, ... } Start rules phase
    memoryDisplay      { grid, items, ... }          Show memorization content
    memoryQuestion     { question, options, ... }    Show question (non-match)
    memoryReveal       { correctAnswer, ... }        Reveal correct answer
    memoryRecap        { standings, history }        Round recap
    memoryFinalResults { winner, standings }         Game over
    memorySync         { phase, ... }                Full state sync
    memoryAnswerReceived { playerName, ... }         Player answered

KEY STATE VARIABLES (MemoryGame.jsx):
  - phase: 'rules'|'display'|'match'|'question'|'reveal'|'recap'|'speedRound'|'finalRecap'
  - challengeType: 'match'|'missing'|'difference'|'sequence'
  - matchGrid: Array of items for match game
  - matchedPairs: Indices of matched cards
  - flippedCards: Currently flipped card indices (max 2)
  - canFlip: Whether player can flip cards
  - matchPoints, matchMistakes, matchComplete: Match game state

TIMER SYNC (same pattern as Trivia/Quick Math):
  - Server sends absolute timestamps (rulesEndTime, displayEndTime, etc.)
  - Client calculates remaining time using getServerTime()
  - Fallback sync request if phase doesn't transition after timer expires

EXIT BUTTON:
  - Fixed position top-right, visible to game master only
  - Allows returning to room during game

SPEED ROUND (Round 4 - Sequence):
  - Items shown one at a time with sequenceItemTime delay
  - 2 second delay after last item before tiles disappear
  - Players tap items in correct order
  - Points based on correct sequence length + time bonus

SERVER FUNCTIONS (sockets.js):
  generateMatchChallenge()      Generate pairs grid for match game
  generateMissingChallenge()    Generate missing item challenge
  generateDifferenceChallenge() Generate spot the difference
  generateSequenceChallenge()   Generate sequence for speed round
  startMemoryRound()            Initialize round with rules phase
  startMemoryGroupTurn()        Start display phase for difficulty group
  showMemoryQuestion()          Transition to question phase
  revealMemoryAnswer()          Show correct answer, calculate points
  advanceToNextMemoryGroup()    Move to next difficulty group
  showMemoryRecap()             Display round recap
  startNextMemoryRound()        Advance to next round or end game
  endMemoryGame()               Final results with winner

INSTRUCTIONS (CHALLENGE_INSTRUCTIONS):
  Each challenge type has: title, description, steps[]
  Displayed during rules phase before each round
